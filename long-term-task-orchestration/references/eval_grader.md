# Eval Grader: Generated Artifact Quality Assessment

After generating all Skill files, launch an independent eval agent to review artifact quality. The eval agent is a complete agent session launched via `claude --print`, with full tool capabilities including file reading and command execution, and can autonomously decide what methods to use for inspection.

---

## Flow

```
Main Agent completes Step 4 (SKILL.md written, all scripts generated)
    ↓
Build grader prompt → write to temporary file
    ↓
claude --print launches eval agent (independent session, --cwd pointing to generated skill directory)
    ↓
Eval agent inspects all files autonomously → writes assessment results to eval-report.json
    ↓
Main Agent reads eval-report.json
    ↓
Findings present?
  → Yes: abstract and generalize problems → fix → launch another eval agent round
  → No: pass, proceed to Step 5
    ↓
Maximum 3 rounds. Problems remain after round 3 → present remaining issues to user
```

---

## Grader Prompt Template

The following template is assembled by the main Agent before initiating eval. Replace `{skillDir}` with the absolute path to the generated Skill directory, and `{reportPath}` with the absolute path to the assessment report.

````
You are an independent quality reviewer responsible for inspecting all files in a newly generated long-running task Skill.
Your goal is to find hidden problems that would cause this Skill to fail in actual execution, produce poor results, or mislead the Agent that uses it.

The Skill directory is located at: {skillDir}

Please read all files in the directory (SKILL.md, scripts/*.js, references/*.md) and inspect them using whatever methods you see fit.
The following are key dimensions to focus on, but not limited to — if you find other issues, please report them too.

## Inspection Dimensions

### 1. Hardcoded Paths
Do any scripts contain hardcoded absolute paths (e.g., `/Users/xxx/...`, `/home/xxx/...`).
All paths should be read from command-line arguments (`--root`) or `.agent.env`, and should not assume a specific directory structure.
Scope of inspection includes script code and prompt text generated by build-prompt.

### 2. Instruction Tone
In instruction text targeted at the Agent (SKILL.md, phase references, prompts generated by build-prompt),
are there large numbers of imperative phrases like "must", "forbidden", "shall not" without explaining the reason.

The Agent is not a command-executing robot — it needs to understand **why** a constraint must be followed to make correct judgments at edge cases.
Good constraint phrasing is "do X, because not doing so leads to Y", not "must do X".

Note: this doesn't mean imperative language can never be used; rather, every imperative constraint should be accompanied by a reason.
If a constraint's reason is obvious (e.g., "output must be valid JSON"), a brief reason suffices.
If the reason is not obvious, the absence of an explanation is a finding.

### 3. Script Executability
For each .js file in the scripts/ directory, attempt `node --check <file>` to verify syntax validity.
If syntax errors are found, report the specific error messages.
Also check whether inter-script references are correct (e.g., the path used by dispatch.js to require build-prompt.js).

### 4. Contract Consistency
The artifact contract paths used to determine key state transitions must be consistent, to avoid validation failures due to output path mismatches that waste work.

### 5. Self-Containment
Are prompts built by build-prompt.js truly self-contained:
- Do they include the complete list of file paths (not just placeholders)
- Do they clearly specify the complete output file path
- Do they include an example of the output format
- Do they explain the validation criteria (what counts as success)
Subagents are fresh sessions with no knowledge of any context; the prompt must not assume it "already knows" any information.

### 6. Idempotency
When discover.js is run repeatedly, will it overwrite already-completed task states.
Correct behavior is: load existing manifest → preserve completed entries → only process new or changed entries.

### 7. Other
Any issues you consider worth reporting. Trust your judgment — if a design makes you think "this might cause problems when actually running", report it.

## Output Requirements

Write the assessment results to `{reportPath}`, in JSON format:

```json
{
  "pass": false,
  "findings": [
    {
      "dimension": "Hardcoded Paths",
      "severity": "error",
      "location": "scripts/dispatch.js:47",
      "description": "Output file path is hardcoded as /tmp/task-data/..., should be derived from --root parameter",
      "suggestion": "Change to path.join(root, '.task-data', 'segments', ...)"
    }
  ],
  "summary": "Found 2 errors and 1 warning. Main issues are in path handling and prompt self-containment."
}
```

Severity has two levels:
- **error**: Will directly cause runtime failure or incorrect output (e.g., path mismatch, syntax error)
- **warning**: Won't directly cause failure, but will reduce effectiveness or maintainability (e.g., imperative language without reasons, missing examples)

If no issues are found, output `{ "pass": true, "findings": [], "summary": "No issues found" }`.

**Do not report style preferences or nice-to-have suggestions. Only report findings you believe will actually cause problems.**
````

## Main Agent Principle for Processing Feedback: Abstract Generalization

After receiving the eval report, the main Agent follows one core principle when fixing: **do not patch individual findings; instead, identify the pattern behind the finding and apply a generalized fix.**

### Example

**Eval report**:
> dispatch.js line 47 hardcodes `/tmp/task-data/segments/`

**Incorrect fix**: only change line 47.

**Correct fix**:
1. Identify the pattern: multiple places in scripts may have path concatenation not based on the `root` parameter
2. Globally inspect all path concatenation in all scripts
3. Uniformly change to derive from the `--root` parameter
4. If the prompt generated by build-prompt has similar issues, fix those too

### Example

**Eval report**:
> "Must not use Agent tool nested calls" in SKILL.md is missing a reason explanation

**Incorrect fix**: only add a reason to this one constraint.

**Correct fix**:
1. Identify the pattern: all imperative constraints in all Agent-facing instruction text should be checked for accompanying reasons
2. Scan all imperative language in SKILL.md and all phase references
3. Uniformly add "because..." reason explanations

### Generalization Judgment Standard

For each finding, ask yourself: **could this problem also exist in other locations?** If yes, do a global sweep before fixing — do not only fix the location pointed out by eval.

---

## Loop Termination Conditions

- Eval agent returns `"pass": true` → terminate, proceed to Step 5
- 3 eval rounds completed → terminate, present remaining findings to user
- Two consecutive rounds have identical findings (fix did not take effect) → terminate, present to user and explain that fix attempts were unsuccessful
