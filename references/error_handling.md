# 错误处理规范

本文档规定长程任务 Skill 的运行时错误处理策略。设计决策（状态存储、分组等）见 `architecture.md`。

---

## 核心原则：错误在最小范围内解决

"最小范围"是分层的：

- **子任务内闭环**：subagent 执行过程中发现产出不合法（如编译不通过、JSON schema 校验失败），应在当前会话内先行修复，而不是把错误留给后续步骤。修不好时 revert 工作区、标记 FAILED，明确告知上层，不让有问题的产出混入完成队列。
- **阶段内收敛**：Phase 2（批量执行）阶段的失败任务必须在本阶段处理完毕（重试或标记 FAILED），进入 Phase 3（收尾验证）时只允许存在"已完成"或"明确标记 FAILED"两种状态。阶段之间的交接必须是干净的，否则下游基于问题产出继续工作，最终在验收阶段才暴露，浪费整条链路。

## 成功判定

检查 Task Contract 约定的产出文件：文件存在 + 格式合法（如 JSON.parse 通过）= 成功。

不要解析 subagent 的文本输出——文本不可靠（可能被截断、格式不一致）。

## 三层重试机制

**内层：恢复会话（续传）**

适用场景：进程崩溃、网络异常、compress 错误等导致的异常退出，任务本身没有错。

dispatch 记录每次执行的 `conversationId`。检测到异常退出（`execAsync` reject 且产出文件不存在）后，用 `zulu run -c <conversationId>` 恢复同一会话，发送"继续"让 Agent 从断点接着跑。

```javascript
// 内层：恢复会话（在 runChunk 的 catch 分支中，仍使用 execAsync 保持异步）
const resumeCmd = [
    `zulu run`,
    `--ptoken=${shellQuote(ptoken)}`,
    `-m=${shellQuote(model)}`,
    `-c=${shellQuote(lastConversationId)}`,  // 恢复同一会话
    `--query="继续上次任务"`,
    `--cwd=${shellQuote(cwd)}`
].join(' ');
await execAsync(resumeCmd, { timeout: 30 * 60 * 1000, maxBuffer: 50 * 1024 * 1024 });
```

这是成本最低的重试方式——Agent 不需要重新理解任务，直接从中断点继续。注意重试同样使用 `execAsync`（见 `script_patterns.md` § dispatch.js），保证不阻塞并发池中其他任务的执行。

**中层：带反馈的新会话（针对性修复）**

适用场景：subagent 正常完成，但产出校验失败（tsc 报错、AST diff 不为空、JSON schema 不合法）。

开一个全新 subagent 会话，将完整的错误信息（报错内容、失败的校验规则）作为上下文附加进 prompt。Agent 只修错误点，不重新处理整组文件。

- 最多重试 2-3 次
- 每次附加累积的错误信息（不只是最近一次），帮助 Agent 理解反复失败的模式
- 超过上限：revert 工作区（删除或还原已修改的文件）、清理环境、标记 FAILED
- 到此为止，子任务层面不再做更多尝试

**外层：主 Agent 重新调度（决策层）**

适用场景：中层耗尽后，积累了一批 FAILED 任务。

主 Agent 检查 FAILED 数量和错误特征，做出决策：

- **FAILED 数量少（如 ≤5%）**：错误可能是偶发的，值得为这批文件重新 dispatch 一轮（重新分组、生成新 prompt、启动新 subagent），相当于把它们当作全新任务重跑。
- **FAILED 数量多（如 >10%）**：可能说明任务规则本身有问题（规则描述不清楚、目标文件有系统性问题），盲目重跑只是浪费 Token。主 Agent 应先分析失败模式，可能需要调整 build-prompt 的规则描述，或排除特定文件。

需要你来判断是否值得重试。不可修复的错误无需重试。

### 三层对照表

| 层级 | 触发条件 | 实现方式 | 成本 | 上限 |
|------|---------|---------|------|------|
| 内层 | 进程崩溃/网络异常，产出文件不存在 | `zulu run -c` 恢复同一会话 | 最低（续传） | 1-2 次 |
| 中层 | 产出存在但校验失败 | 新会话 + 完整错误信息 | 中 | 2-3 次 |
| 外层 | 中层耗尽后，批量 FAILED | 主 Agent 决策重新 dispatch | 高 | 主 Agent 判断 |

## IN_PROGRESS 残留处理

Agent 被中断时可能有任务停留在 IN_PROGRESS。新会话恢复时需要处理：

- **方案 A（重置为 TODO）**：适用于操作幂等、无副作用的任务（如只读审查、可回退的迁移操作）
- **方案 B（检查产出文件决定）**：适用于修改源文件的任务。检查产出文件是否存在且合法，存在则标记 DONE，否则重置为 TODO

选哪个方案取决于"重复执行这个子任务是否安全"。只读任务选 A；会修改文件且产出可校验的选 B。
